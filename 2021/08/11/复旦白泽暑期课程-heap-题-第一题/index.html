复旦白泽暑期课程 heap 题 第一题

ctf
  • ctf-pwn
  • heap

前情分析

拿到这个题，显然是个标准的菜单题。

那么我们需要看看各个功能里面，具体的操作是什么样的。

ida里打开看到

  • show_item 函数可以输出一个堆块中的内容

  • add_item 可以新分配一个堆块

  • change_item 可以修改一个已经存在的堆块里面的内容，这里没有限制可以修改多少，因此有一个任意字节堆溢出的漏洞。

  • remove_item 可以删除一个堆块

除了每个函数的功能之外，这个题目有另外需要注意的几点：

  • 首先，在给堆块读入数据的时候，读完之后会自动给下一个地址增加一个0，例如下图所示add功能中所示。

  • 然后是，在remove功能中，其给free之后的指针置0，避免了直接的UAF

  • 最后是需要注意分析寸堆块的结构长什么样（即itemlist和qword_2020A8到底是什么）。

    首先我们看看add功能里面，v2显然代表了需要分配的堆空间大小，那itemlist的低4字节显然就是存这个堆块的大小了。然后qword_2020A8显然是malloc之后的指针列表。

    然后我们再来看看，再bss段里面这两个东西的分布情况。

    从图中可以看到，itemlist 后面8个字节就是qword_2020A8。那么我们再看每次分配的时候，itemlist和qword_2020A8的指针增长趋势可以看到，itemlist每次增长4个4字节，也就是2个8字节，那么刚好就跳到了qword_2020A8 之后8个字节的地方，从图上看就是2020E0的地方。此时我们暂时把这个地方标记为itemlist1

    然后再看qword_2020A8的增长，以此是长2个8字节，跟itemlist是一样的，所以它相当于跳到了itemlist1之后8个字节处，也就是2020E8的地方，正好不会与itemlist1所用的8字节空间重合。

    因此分析到这里我们就可以很明显知道了，itemlist和qword_2020A8的作用，一个是存堆块分配的空间，一个是存堆块分配的指针。相当于这两个是一个结构体中的部分。

终于至此，我们分析完了这个题目程序的基本架构。

结果如下：

开搞

有了上面的分析之后，我们就可以开始思考怎么来做这个题了。

首先，我们这里基本上可以考虑是找个libc的one_gadget来做

因为我刚入门，只能想到这个做法，不然再这么多保护的情况下，在好像还没法弄到栈里的前提下，好像不知道怎么弄ROPgadget来做。

堆做法的合理性的思考大致如下

1、首先我们明确了需要libc的one_gadget，那么我们需要leak libc的基址，这个可以利用unsorted bin里面的fd和bk指针搞定，因为当只有一个堆块在unsorted bin中的时候，这个fd和bk是指向bin[0]的地方，也就是main arena + 88。因此这个leak思路暂时可行的。

bin[0]就是unsorted bin。

2、有了one_gadget的基址之后，就是怎么才能改变控制流。我们知道一个叫做__malloc_hook的函数，它的调用条件是在malloc调用的时候看看__malloc_hook中有没有函数的地址。因此我们可以考虑把libc的one_gadget地址写到这里，这样就可以在malloc的时候开shell了。

3、如何改写__malloc_hook就比较重要了。目前能想到的就是unlink，但是这个办法我测了一下有点问题，暂时不知道为什么（可能是libc2.23里面也会检查看fd出去再bk回来是不是自己么？）。然后我找了很多资料，发现其实可以利用fast bin的fd指针，让它指向__malloc_hook，然后在分配一个fast bin对应大小堆块的时候，可以把__malloc_hook所在的地址当成一个堆块分配出来，然后就可以在里面修改对应的信息了

真实处理的思路 - exp

有了上面的分析之后，我们就可以开始按照这个流程写exp了

exp的具体思路如下：

  • 先对程序各个功能建模，如下所示：

  • 然后我们要leak libc。这个步骤没有想象中简单。

    错误示范：本来想先free掉一个大于0x80（加上meradata后的堆大小）的堆块，让其进入unsorted bin，然后再malloc出来，并给content内容为””一个空字符串，这样show()出来就能得到其内容，然而这个是不行的。我以为是“”空字符串的问题，会引入\x00，所以我就把前8个字节（fd）写成8B的字符a，然后一连串输出到后8字节（bk），结果居然后面8个字节的最低1个字节也被写成\x00了。于是我去找问题，在ida里面就看到了，读入之后，会将读入后的末尾加上一个0（在前情分析里提到过了）。可见这个程序就是防止你去这么leak，🐂。

    正确思路：这样的话我们就要想个别的办法来leak了。找了很多资料之后，我弄明白了。这个思路很巧妙，首先我们先分配一个小堆块（加上metadata就0x20就够），然后再分配两个堆块（要求这两个堆块加起来大小（包括他们的metadata）大于0x80就好）。

    然后我们修改第一个小堆块，目的是让它把第二个堆块的头修改掉。

    这就是为什么第一个小堆块我说0x20就行，因为开大了没有必要

    我们需要把第二个堆块的头改成 00 和 0xnn，这里nn是第二个和第三个堆块合起来的大小（包括metadata），这样是为了伪造二三个堆块为一个堆块。

    然后再把第二个堆块free掉，这样伪造的这个二三合体堆就会被放到unsorted bin里（所以之前说这两个合起来大小要大于0x80）。

    而且此时，第三个堆块的指针还在，也就是unsorted bin的后半段其实是从外面可以访问了。

    然后我们再申请一个跟最开始第二个堆块一样大的堆块，这个时候就会从unsorted bin里面分割出一部分分配，这就使得剩下的部分刚好就是原来的第三个堆块所指的位置，并且这个时候，它的fd和bk都是main arena + 88。

    这样我们不需要写入数据，就有了想要的地址，并且不用再分配了，因为指针一直都在，没有free过。

    然后只要把第三个堆块的内容数据就可以leak出libc的地址了

    注意：这个地方还要记得再第三个堆块后面再申请一个小堆块，防止free伪造二三合体堆的时候，跟top chunk合并了。

    注意2：这里我们还需要main arena + 88的offset，这个其实很简单，gdb一下程序，然后找到libc的基址（Pwngdb（不是pwndbg噢~~~初学很容易看混的）在gdb的基础上给了libc指令，可以直接看），然后复制出来，再找到此时main arena + 88的真实地址，然后两个相减就是offset了。

  • 有了libc之后，我们就是要改__malloc_hook了。

    错误思路：用unlink 来使得__malloc_hook的地址下写入one_gadget 。也有一种可能，就是我unlink理解不到位，到底要写什么到fd和bk写的有问题，这个可以再去考究一下。

    正确思路：我们需要弄一个fast bin里的堆块，然后对里面的内容进行修改。那么怎么能够对bin里面的，也就是free的堆块进行写操作呢？在这个题把指针还都清空的情况下。

    显然我们可以继续用上面leak的思路。我们现在在unsorted bin中的一个堆块就是一个指针还可以用的堆块。正好大小是0x70（算上metadata），因此我们先拿出来再free掉，然后它就进到fast bin了。

    这个时候，我们之前的第三个堆块（下标为2）的指针就指向fast bin里面的那个内容。

    然后我们修改第三个堆块（下标为2）的fd字段的位置，改成__malloc_hook的地址。

    这样我们再分配一个的时一样大小的（我这个做法里，他们是串在fast bin中0x70字段，所以我malloc 0x60）。

    这样这个时候，fast bin里面还有一个块，这个块的地址是__malloc_hook，然后我们继续malloc 0x60，它则还会再fast bin的0x70字段去找，就会找到这个__malloc_hook以及其后面的0x70（前0x10是metadata）作为堆块，给其下标为5。

    重点中的重点：这里要注意，其实我在修改fast bin那个堆块的fd位置的时候，不能写__malloc_hook，因为首先我们知道前面有0x10是metadata，所以我们至少要把fd写为__malloc_hook - 0x10（因为目的是让__malloc_hook的地方是one_gadget）。但是这么写还是不行，分配不出来，会报错，然后我一直测试，发现就是__malloc_hook - 0x23成功了，可以分配出来，并且没有报错。看了很多的资料，大家也都是0x23，原因暂时还不知道。

    重点中的重点的重点：那么这样分配出来之后，在没用的地方放杂物，最后__malloc_hook的位置放one_gadget真的就行了吗？？？答案是否定的，因为one_gadget使用条件的原因，在这个情况下，4个one_gadget都不行…….然后我找了很多资料，有人说可以把控制流改成 malloc -> __malloc_hook -> realloc -> __realloc_hook 因为realloc前面有很多push操作，可以控制栈的高度。

    下面我们来讲讲这个操作要怎么写

    首先我们找到realloc的offset，然后加上基址就好了。那么要实现上面的流程，我们要将__malloc_hook里面写成realloc的地址，然后在__realloc_hook里面再写上one_gadget。

    注意：__realloc_hook 在 __malloc_hook - 0x8 的地方。

    然后这个做法我没有仔细去调栈的内容，就直接这么写用第二个one_gadget就拿到shell了。不过实际上，可以通过改变__malloc_hook中写的地址，来跳过realloc中的一些push操作，以此达到操作栈里面内容的效果。具体可以看看这个博客，主要讲的one_gadget的限制和利用

  • 最后：我们最后再调用一下malloc函数就能拿到shell了
